# Karpathy-Inspired Coding Agent Guidelines

## 核心原則

目標是減少非必要的大改、誤解需求、修壞既有功能及過度工程化。

簡單 typo 或明顯 one-line fix 可以快速處理；涉及資料、排程、庫存、
API、資料庫、UI 流程、權限或計算規則時，必須遵守以下流程。

## 1. 先理解，再寫程式

修改前先確認：

- 使用者真正要解決的問題。
- 受影響的檔案、函式、資料表與 Excel 欄位。
- 不可改變的既有流程，例如 endpoint、schema、路由與 Excel 格式。
- 是否存在多種解讀；若有，列出假設，不要默默選擇。

禁止未讀完相關檔案便修改、因單一錯誤進行大範圍重構，或未確認需求便改欄位名稱、資料格式及 API response。

## 2. 優先小改

- 只改與需求直接相關的檔案。
- 保留原本可運作的功能。
- 不因風格偏好重排大量程式碼。
- 不把簡單問題擴張成大型架構重構。

除非使用者明確要求，否則不要重寫整個 module、改變專案結構、更換框架，或移除可能被其他流程依賴的程式。

## 3. 找最小可驗證路徑

1. 找到可重現問題的輸入或操作。
2. 明確定義預期結果。
3. 修改最小範圍。
4. 執行測試或 smoke test。
5. 回報修改內容與驗證方式。

無法完整測試時，至少執行適合的低成本檢查：

- Python：`python -m py_compile file.py`
- 前端：`npm run build` 或 `npm run lint`
- Flask/API：啟動後以 `curl` 測試主要 endpoint
- Excel：開啟產出檔並檢查 sheet、欄位、格式與數值

## 4. Bug 先重現，再修正

- 先找最小重現案例，不憑猜測修改。
- 資料問題保留一筆範例作為驗證。
- 修正後確認原錯誤已消失。
- 不以空的 `except Exception: pass` 隱藏錯誤。
- 未驗證時不可宣稱已修好。

## 5. 不隱藏不確定性

資訊不足時，明確說明已知、未知、採用的假設及需要使用者確認之處。不要假裝已知道資料表、API、欄位或業務規則。

## 6. 避免過度工程化

優先使用簡單、可維護、容易 debug 的方案。小需求不要加入不必要的抽象層、class hierarchy、configuration、plugin system 或未使用的 helper。

## 7. 保留既有規則與資料格式

- Excel 格式、欄寬、顏色與 sheet 名稱盡量固定。
- 人工計算與系統計算的欄位定義不可混淆。
- 比對邏輯必須可追蹤、可重跑。
- 特殊品項規則不可被一般 normalize 覆蓋。

Bead 比較工具規則：

- 舊檔為人工計算；新檔為系統計算。
- 差異為系統減人工。
- 兩劑品項：需求取大、庫存取小。
- `QTG != TG`。
- `T4 DB != T4`。
- `AMY-A = AMY`。
- `CREA` 是組合測試，不單獨拉出 `tCREA-U搭配量`。
- `tCREA-U` 僅為 CREA 組合必要搭配 bead，不列為獨立需求品項。

## 8. 完成後回報

說明修改檔案、主要變更、驗證方式、產出檔案，以及尚未驗證或需要注意的事項。不要只回覆「已修正」。

## 9. Excel 與報表

- 保留模板 sheet 順序、標題、底色、文字顏色、欄寬、freeze pane 與 filter。
- 數字欄使用數字格式；差異欄清楚標示正負。
- Summary 必須能追溯至明細，規則說明放在獨立 sheet。
- 優先複製模板再更新資料，不重建不同外觀的報表。

## 10. 資料庫與 API

- 不任意更改既有欄位名稱或破壞 endpoint response。
- 新欄位必須向下相容。
- 寫入前確認 `key`、`lot_no`、`work_order_no`、`device_sn` 等識別欄位。
- 重要寫入保留 log 或 audit trail。

## 11. UI

- 除非需求明確要求，不修改後端計算核心。
- 不破壞 button、API call、form submit 與 scan flow。
- 手機畫面減少滑動，主要操作盡量在同頁完成。

## 12. 最後檢查

- [ ] 是否真正解決使用者提出的問題？
- [ ] 是否存在不必要的大改？
- [ ] 是否保留既有功能？
- [ ] 是否執行測試或基本驗證？
- [ ] 是否說明修改內容？
- [ ] 是否標出未驗證部分？
- [ ] 是否產出使用者需要的檔案？

一句話：先理解，少改動，可驗證；不要猜，不要大改，不要假裝確定。
